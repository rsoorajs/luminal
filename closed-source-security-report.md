# luminal-cuda Closed-Source Security Report

## Attack Surface Inventory & Mitigation Plan

---

### 1. Egglog Optimization Rules — Plaintext in `.rodata`

**Vulnerability:** All 9 `.egg` files plus programmatic rules in `logical.rs` are embedded as UTF-8 string literals via `include_str!` and `Rule::raw()`. They survive symbol stripping entirely — `strings _native.abi3.so | grep "(rule"` recovers them. This is 43K+ of proprietary fusion logic: FlashInfer attention matching, cuBLAS dispatch heuristics, sigmoid/exp recognition, scatter elimination, batch-collapse rules.

**Fix A — Build-time rule compilation:** Write a `build.rs` proc-macro or code generator that parses the `.egg` files at compile time and emits Rust match arms / function calls instead. The egglog interpreter still runs, but the rule *text* never appears in the binary — only compiled egglog `Program` structs. No string survives.

> **Danger:** This is a major refactor. The egglog crate's API takes string programs — you'd need to either (a) use the egglog Rust API to construct `Command`/`Rule` structs directly without going through the parser, or (b) fork egglog to accept pre-parsed ASTs. If rules change frequently during development, the iteration loop gets slower. Also, any egglog version upgrade could break the codegen.

**Fix B — Runtime string obfuscation:** XOR or compress the rule strings at compile time (via a proc-macro wrapping `include_str!`), decompress/decrypt at runtime before passing to egglog. Simple, low-effort.

> **Danger:** This is security-by-obscurity — a determined attacker sets a breakpoint on egglog's parser entry (even without symbols, they can find it via the egglog string constants that are in the *open-source* crate) and reads the decompressed string from memory. It raises the bar from `strings` (5 seconds) to a debugger session (30 minutes). Also adds ~1ms startup latency per rule decompression.

**Fix C — Move rule matching to native Rust (no egglog for proprietary rules):** Implement the proprietary fusion patterns (FlashInfer attention, batch matmul, scatter elimination) as Rust graph-pattern-matching passes that run before/after egglog, rather than as egglog rules. The open-source cuBLAS rules stay in egglog; the secret ones become opaque compiled code.

> **Danger:** You lose the elegance of equality saturation for these rules — they become fixed-order passes that can't interact with egglog's cost model or discover emergent optimizations from rule composition. You'd also need to maintain two optimization frameworks (egglog + custom passes). If the proprietary rules have interactions with open-source rules (e.g., scatter elimination enabling a cuBLAS fusion), you'd need to handle cross-system dependencies manually.

### 2. CUDA Kernel Source — Plaintext `format!()` Strings

**Vulnerability:** 18+ CUDA kernels are built via `format!()` in `hlir.rs` and `other_ops.rs` and compiled at runtime via NVRTC. The full C++ source (`__global__ void add_k(...)`, reduce kernels, gather/scatter, etc.) is readable with `strings`. 44 kernel signatures found.

**Fix A — PTX embedding:** Compile the CUDA kernels to PTX at build time (via `nvcc` in `build.rs`) and embed the PTX instead of the source. NVRTC can load PTX directly. PTX is a low-level assembly — technically reversible but orders of magnitude harder to read than C++.

> **Danger:** PTX is architecture-specific — you'd need to either (a) ship multiple PTX variants for different SM architectures, or (b) ship SASS (even more opaque but locked to a single GPU arch). The current `format!()` approach generates kernels with dynamic shapes/dtypes, so you'd need to pre-instantiate all dtype×shape combinations or build a PTX template system. Also, PTX stability across CUDA toolkit versions is not guaranteed.

**Fix B — Obfuscate kernel source strings:** Same XOR/compress approach as the egglog rules. Low effort, modest protection.

> **Danger:** Same as 1B — a breakpoint on `nvrtc::compile_ptx` captures the plaintext source. These kernels are also relatively standard (elementwise ops, reductions) — an experienced CUDA developer could rewrite them from the kernel *names* alone without needing the source.

**Fix C — Ship pre-compiled CUDA fatbinary:** Compile all kernels at build time into a fatbinary (`.cubin` with multiple SM targets) and load them via `cuModuleLoad` instead of NVRTC. No source text survives.

> **Danger:** Loses the dynamic shape/dtype templating — you'd need to pre-instantiate a matrix of (op × dtype × potentially shape-dependent params). The fatbinary grows large. You also lose support for GPU architectures released after the build. This is how cuDNN works, but cuDNN has a dedicated team managing the combinatorial explosion.

### 3. SBOM Disclosing Build Environment & Dependency Graph

**Vulnerability:** `luminal_cuda_python.cyclonedx.json` (auto-generated by `cargo-cyclonedx` via maturin) ships in the wheel and contains:
- Private repo paths: `file:///home/ubuntu/luminal/...`
- Exact pinned commits: luminal `f32161d`, egglog `0a8cc35`
- 394 dependency components with exact versions
- Build timestamp: `2026-04-02T00:13:36Z`

An attacker can clone the exact open-source luminal commit and diff against the wheel to isolate closed-source additions.

**Fix:** Exclude the SBOMs from the wheel. In `pyproject.toml`:

```toml
[tool.maturin]
exclude = ["**/sboms/**"]
```

Or delete the files in a post-build step before packaging.

> **Danger:** Essentially none. SBOMs are useful for compliance audits and vulnerability scanning, but they can be generated on-demand internally — they don't need to ship to customers. If a customer *requires* an SBOM (some enterprise procurement does), you can provide a redacted version that lists dependencies without file paths or git hashes.

### 4. Panic Strings Leaking Source Paths

**Vulnerability:** 277 strings containing `/home/ubuntu/` survive in the binary. These come from `panic!()`, `unwrap()`, `assert!()` etc. in both your code and dependencies. They reveal the build machine username, the private repo location (`/home/ubuntu/luminal/`), the exact egglog fork commit, and the Rust toolchain version.

**Fix:** Add `--remap-path-prefix` to RUSTFLAGS in the build:

```bash
RUSTFLAGS="--remap-path-prefix /home/ubuntu=. --remap-path-prefix /home/ubuntu/.cargo=cargo" \
  maturin build --release
```

> **Danger:** Minimal. Panic messages become relative paths (e.g., `./luminal/src/runtime.rs`) which are still useful for debugging. The main risk is if you remap too aggressively and make crash reports unintelligible — but since these are distributed binaries, your users don't debug panics by source line anyway. One caveat: some crates embed paths at build time via `env!("CARGO_MANIFEST_DIR")` — remap won't catch those (but we already removed those from jit.rs).

### 5. FlashInfer `wrapper.cu` and CUTLASS Headers Shipped as Package Data

**Vulnerability:** The wheel bundles `flashinfer_data/wrapper.cu` (357 lines, 14K) and 1,010 CUTLASS header files (~31M). The wrapper reveals FlashInfer kernel instantiation patterns, attention type configurations (causal mask, no sliding window, no logits soft cap), and the fp32-fp16 casting strategy for prefill.

**Fix A — Pre-compile FlashInfer at build time:** Compile `wrapper.cu` into `.so` files for all supported SM architectures (sm_80, sm_86, sm_89, sm_90) at build time. Ship the pre-compiled `.so` files instead of the source + headers. Remove the JIT path entirely.

> **Danger:** You lose support for future GPU architectures without a wheel rebuild. The current JIT approach is a feature — it adapts to whatever GPU is present. You'd need to maintain a build matrix (SM arch x HEAD_DIM combinations). The wheel size might actually shrink though (4 small `.so` files vs 31M of headers). The biggest risk: if a user has an SM architecture you didn't pre-compile for, FlashInfer silently fails. You'd need a fallback path.

**Fix B — Ship only the compiled `.so` for common archs, keep JIT as fallback:** Pre-compile for sm_80/86/89/90 and ship those. Only JIT-compile (requiring the source) for unrecognized architectures. Ship the source but obfuscated/compressed.

> **Danger:** The source is still technically present (for the fallback path), just harder to extract. A hybrid that's more complex to maintain. The happy path (pre-compiled) is secure; the fallback path is not.

**Fix C — Accept the exposure.** FlashInfer is open source. The wrapper.cu instantiation is specific to luminal's calling convention but reveals minimal proprietary logic beyond "we use FlashInfer with these type parameters." The CUTLASS headers are NVIDIA's open-source code.

> **Danger:** None architecturally. The question is whether the *specific instantiation choices* (which attention variants, which type casts, which tile configs) constitute meaningful IP. If your competitive advantage is the egglog rules that *select* FlashInfer, not the FlashInfer wrapper itself, this exposure is low-priority.

### 6. Op Names and IR Structure in String Constants

**Vulnerability:** Even with symbols stripped, the op type names (`KernelAdd`, `KernelBatchMatMul`, `FlashInferAttention`, `KernelScatterNoCopy`, `Sigmoid`, `CUBLAS_OP_N`, etc.) survive as string constants because they're used at runtime for egglog sort registration, error messages, and tracing. An attacker can reconstruct the full IR taxonomy from these alone.

**Fix A — Rename/obfuscate op names at build time:** A proc-macro that replaces all op name strings with hashes or random identifiers (e.g., `KernelBatchMatMul` -> `op_a7f3`). The egglog rules would use the same obfuscated names.

> **Danger:** Makes debugging and tracing nearly impossible. Every error message, log line, and profiler trace becomes opaque. If a user reports "op_a7f3 failed with CUDA error 700", you need a lookup table to decode it. This is a significant developer experience hit. Also, the egglog rules would need to be obfuscated consistently — a single missed reference and the rules break.

**Fix B — Accept the exposure.** Op names reveal *what* operations exist, not *how* they're implemented or *when* they're selected. The existence of "KernelBatchMatMul" is unsurprising to anyone familiar with ML compilers. The real IP is the fusion rules, not the op vocabulary.

> **Danger:** None. This is a pragmatic trade-off — the op names have low IP value and high debugging value.

### 7. Dynamic Interception via Debugger (Post-Strip)

**Vulnerability:** Even without symbols, an attacker can still intercept egglog rule strings at runtime. The open-source `egglog` crate has its own symbols and known entry points. An attacker can: (a) find the egglog parser entry via the egglog crate's error strings, (b) set a hardware watchpoint on large string allocations, or (c) use `LD_PRELOAD` to intercept `malloc` and filter for egglog-shaped strings. Stripping removed the *easy* path (named breakpoints) but the data still transits through memory.

**Fix A — The egglog rules never exist as strings (Fix 1A above).** If the rules are compiled to native Rust AST construction, there's no string to intercept.

> **Danger:** Same as 1A — major refactor.

**Fix B — Anti-debugging measures:** Detect `ptrace`, check `/proc/self/status` for `TracerPid`, use `prctl(PR_SET_DUMPABLE, 0)`. Refuse to run under a debugger.

> **Danger:** This is hostile to legitimate users debugging their own applications. It's trivially bypassable (patch the check, use a kernel module, or just dump the process memory directly). It signals adversarial intent and damages trust. Not recommended.

**Fix C — Use encrypted IPC to a separate process.** Run egglog in a hardened child process that receives encrypted rules, decrypts them in-memory, runs the optimizer, and returns results. The parent process never has the plaintext rules.

> **Danger:** Massive complexity increase. IPC latency for the large graph data. The child process can still be debugged (just attach to it instead). You're adding an architectural boundary that slows development and provides marginal security gain against a determined attacker. This is the "DRM treadmill" — each protection layer adds cost to you and a small speed bump to the attacker.

### Summary Matrix

| # | Vulnerability | Severity | Best Fix | Effort | Risk to Architecture |
|---|---|---|---|---|---|
| 1 | Egglog rules as plaintext strings | **CRITICAL** | 1A (compile to AST) or 1C (native Rust passes) | High | Medium — changes optimization framework |
| 2 | CUDA kernel source strings | **HIGH** | 2A (PTX embedding) | Medium | Medium — loses dynamic dtype/shape templating |
| 3 | SBOM with paths and git hashes | **HIGH** | Exclude from wheel | Low | None |
| 4 | Panic strings with build paths | **MEDIUM** | `--remap-path-prefix` | Low | None |

**Recommended priority order:** 3 -> 4 -> 1 -> 2 -> 5. Items 3 and 4 are free wins. Item 1 is where the real IP protection happens. Item 2 matters but the kernels are less unique than the fusion rules.
